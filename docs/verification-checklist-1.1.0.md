## Verification gates

### 1. Create form renders in AtoM style

- [ ] Navigate to `/hmt/article/new` in a running AtoM stack.
- [ ] Page renders with AtoM page layout (`layout.php`) and the four collapsible
  sections: Identification, Access points, Description, Administration.
- [ ] Form fields match AtoM `<dl>/<dt>/<dd>` conventions with help text blocks.
- [ ] Submit button reads "Save" and matches AtoM `.c-btn.c-btn-submit` style.
- [ ] No AtoM core template or module file is modified.

### 2. Edit form renders with pre-populated data

- [ ] After a successful create, the redirect lands on `/hmt/article/<id>`.
- [ ] All fields show the values submitted during create.
- [ ] A "Delete" link is present on the edit page and absent on the create page.
- [ ] Page heading includes the article title as a sub-label.

### 3. Submit path persists deterministically

- [ ] POST to `/hmt/article/new` with a valid payload inserts one row in `hmt_article`.
  - `id` is a UUID v4 string.
  - `title`, `source`, `url`, `publication_date`, `summary`, `status` match submitted values.
  - `created_at` and `updated_at` are set to the current timestamp.
- [ ] POST to `/hmt/article/<id>` with changed values updates the existing row.
  - `updated_at` advances; `created_at` is unchanged.
  - All other fields reflect the new submission.
- [ ] Duplicate submit (re-POST same body) does not create a second row (idempotent by id).

### 4. Validation and flash behavior match AtoM conventions

- [ ] Submitting a blank form shows inline error "Title is required." adjacent to
  the title field (not a flash message).
- [ ] Submitting an invalid URL shows inline error "Please enter a valid URL."
- [ ] Submitting an invalid date shows inline error "Please use the date format YYYY-MM-DD."
- [ ] Successful create shows green flash banner: "Article created successfully."
- [ ] Successful edit shows green flash banner: "Article updated successfully."
- [ ] Successful delete shows green flash banner: "Article deleted."
- [ ] Attempting to edit a non-existent id shows red flash: "Article not found."

### 5. Delete flow

- [ ] Clicking "Delete" on the edit page shows a JavaScript confirm dialog.
- [ ] Confirming submits a POST to `/hmt/article/<id>/delete`.
- [ ] The record is removed from `hmt_article`.
- [ ] User is redirected to `/hmt/article/new` with "Article deleted." flash.
- [ ] Cancelling the confirm dialog does not submit.

### 6. Hosted fallback coexistence

- [ ] Existing hosted integration routes (`/plugins/homicide-tracker/…`) respond
  normally with HTTP 200.
- [ ] The plugin bootstrap hook (`php symfony plugin:publish-assets … && php symfony cc`)
  completes without error when `ATOM_BOOTSTRAP_PLUGIN_HOOK` is set.

### 7. Plugin mount sanity

- [ ] The bind-mount in `docker-compose.yml` correctly maps
  `../../atom/plugins/sfHomicideMediaTrackerPlugin` into
  `/atom/src/plugins/sfHomicideMediaTrackerPlugin` (read-only).
- [ ] `atom_worker` service has the same mount so background jobs can resolve routes.
